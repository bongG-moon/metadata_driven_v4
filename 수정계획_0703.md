# 수정계획 0703

## 목적

data analysis flow와 metadata runtime 전달 구조에서 불필요한 payload, prompt token, MongoDB 저장 중복을 줄인다.

중요 원칙:

- MongoDB에 이미 저장된 실제 metadata 원본은 임의로 변형하지 않는다.
- 저장 flow가 만든 원본 구조는 최대한 유지하고, data analysis runtime에서 LLM/prompt에 전달하는 사본만 lean하게 만든다.
- 다운로드 기능에 필요한 원본 rows와 결과 rows는 보존하되, prompt/API/trace에는 preview와 요약만 전달한다.
- 개발자 진단에 필요한 pandas code, trace 정보는 유지하되 답변 생성 LLM 입력에는 compact scope만 전달한다.
- 각 항목은 한 번에 크게 바꾸지 않고, 수정 -> 단위 테스트 -> 대표 질문 검증 순서로 진행한다.

## 1. Metadata 후보 lean copy 적용

### 현재 상태

- `01a_mongodb_domain_metadata_loader.py`
- `01b_mongodb_table_catalog_loader.py`
- `01c_mongodb_main_variable_loader.py`

위 로더들은 MongoDB 문서에서 `_id`만 제외하고 거의 전체 문서를 runtime 후보로 넘긴다.
그 결과 `registration_trace.raw_text`, `raw_trace`, `refined_text`, `review`, `write_result` 같은 authoring/audit 성격 필드가 의도 분석 prompt까지 들어갈 수 있다.

### 수정 방안

- MongoDB 저장 데이터 자체는 수정하지 않는다.
- data analysis flow에서만 lean metadata 사본을 만든다.
- 우선 `01d_metadata_candidates_builder.py`에 공통 sanitizer를 추가한다.
- 제거 후보 필드:
  - `registration_trace`
  - `raw_trace`
  - `raw_text`
  - `raw_text_preview`
  - `refined_text`
  - `review`
  - `write_result`
  - `trace`
  - `existing_matches`
  - `duplicate_decision`
  - `created_by_prompt`
  - `llm_response`
- 단, table catalog의 `payload.source_config.query_template`처럼 실제 조회에 필요한 원문 SQL은 제거하지 않는다.
- `payload.description`, `payload.aliases`, `payload.source_config`, `required_params`, `filter_mappings` 등 의도 분석/조회에 필요한 필드는 유지한다.

### 검증 방법

- metadata candidate builder 단위 테스트 추가:
  - 입력 문서에 `registration_trace.raw_text`가 있어도 출력 metadata에는 없어야 한다.
  - `query_template`은 보존되어야 한다.
  - `_id`는 계속 제거되어야 한다.
- 실행:
  - `python -m compileall -q langflow_components tests`
  - `python -m pytest tests\test_langflow_components.py -q`

### 주의점

- 저장/export/upload 도구의 JSON 원본 보존 정책과 충돌하지 않게 한다.
- MongoDB export 파일에는 원본이 남아도 되지만, LLM prompt 후보에는 들어가지 않도록 분리한다.

## 2. Metadata candidates 중복 구조 축소

### 현재 상태

`01d_metadata_candidates_builder.py`는 아래 구조를 동시에 반환한다.

```json
{
  "domain_items": [],
  "table_catalog_items": [],
  "main_flow_filters": [],
  "runtime_function_helpers": [],
  "metadata_candidates": {
    "domain_items": [],
    "table_catalog_items": [],
    "main_flow_filters": [],
    "runtime_function_helpers": []
  }
}
```

같은 metadata가 top-level과 `metadata_candidates` 내부에 중복된다.

### 수정 방안

- downstream 연결 안정성을 위해 첫 단계에서는 `01d` output shape를 급격히 바꾸지 않는다.
- 대신 `02_intent_variables_builder.py`에서 prompt에 넣을 때는 inner `metadata_candidates`만 추출한다.
- 이후 Langflow 연결이 안정화되면 `01d` top-level 중복 필드 제거를 별도 단계로 진행한다.

### 검증 방법

- `02_intent_variables_builder.build_variables()` 테스트 추가:
  - `metadata_candidates` wrapper가 들어와도 prompt 변수에는 중복 없는 inner candidate만 직렬화되어야 한다.
  - top-level `metadata_load`는 필요 최소 요약만 남기거나 제외한다.

### 주의점

- Langflow 노드 간 연결이 이미 `01D.metadata_candidates -> 02.metadata_candidates_in`으로 되어 있으므로 output 포트 이름은 바꾸지 않는다.

## 3. Function case 단수/복수 중복 정리

### 현재 상태

- `02_intent_variables_builder.py` output schema에 `pandas_function_case`와 `pandas_function_cases`가 함께 있다.
- `04_intent_plan_normalizer.py`는 단수/복수를 모두 받고 `selected_function_cases`까지 추가한다.
- `pandas_execution_plan`의 `apply_pandas_function_case` step에도 같은 선택 정보가 반복된다.

### 수정 방안

- canonical field는 `intent_plan.pandas_function_cases` 리스트로 정한다.
- LLM이 단수 `pandas_function_case`를 반환해도 `04`에서 리스트로 변환한다.
- 정규화 이후 payload에서는 `pandas_function_case` 단수 필드를 제거한다.
- `selected_function_cases`는 payload에 저장하지 않는다.
- `15_pandas_variables_builder.py`와 `17a_pandas_repair_variables_builder.py`는 필요 시 `pandas_function_cases`와 `pandas_execution_plan`에서 계산한다.
- `pandas_execution_plan`의 `apply_pandas_function_case` step은 실제 실행 순서 표현이므로 유지한다.

### 검증 방법

- normalizer 단위 테스트:
  - LLM이 단수만 반환하면 `pandas_function_cases` 리스트 1개로 변환된다.
  - LLM이 복수만 반환하면 그대로 유지된다.
  - 단수/복수가 같은 내용을 동시에 반환하면 중복 제거된다.
  - 정규화 결과에 `pandas_function_case`와 `selected_function_cases`가 없어야 한다.
- pandas variable builder 테스트:
  - function case 선택 정보가 여전히 16번 prompt에 전달되어야 한다.

### 주의점

- 기존 prompt 문구와 specialized guide에서 단수 필드를 안내하는 부분도 함께 갱신한다.
- 한 번에 prompt와 normalizer를 모두 바꿔야 LLM 출력/정규화 계약이 어긋나지 않는다.

## 4. 04 의도 계획 정규화기 중복 함수 제거

### 현재 상태

`04_intent_plan_normalizer.py`에 같은 이름의 함수가 두 번 정의되어 있다.

- `_ensure_function_case_step`
- `_selected_function_cases`

Python에서는 뒤쪽 정의만 살아남기 때문에 앞쪽 함수는 죽은 코드다.

### 수정 방안

- 앞쪽 중복 정의를 삭제한다.
- 남기는 함수는 3번의 canonical `pandas_function_cases` 계약에 맞춰 단순화한다.
- 함수명과 책임:
  - `_function_case_items`: 단수/복수 입력을 받아 리스트로 정규화
  - `_ensure_function_case_steps`: execution plan에 필요한 step만 보강
  - `_dedupe_cases`: function_name, key, input_text, source_alias 기준 dedupe

### 검증 방법

- `python -m compileall -q langflow_components\data_analysis_flow\04_intent_plan_normalizer.py`
- 관련 pytest 실행

### 주의점

- 공정명 단독 fallback 제거 같은 project-specific 방어 로직을 넣지 않는다.
- function case 선택은 prompt/metadata 계약에서 해결한다.

## 5. Preview rows 축소

### 현재 상태

source retriever와 reference runtime의 preview rows 기본값이 20이다.

- `08_dummy_data_retriever.py`
- `09_oracle_query_retriever.py`
- `10_h_api_retriever.py`
- `11_datalake_retriever.py`
- `12_goodocs_retriever.py`
- `reference_runtime/retrieval.py`
- `17_pandas_code_executor.py` trace preview
- `21_answer_message_adapter.py` result table preview

### 수정 방안

- source retriever preview는 5행으로 줄인다.
- pandas execution trace preview는 5행으로 줄인다.
- 사용자 답변 표시는 기존 사용성에 따라 10행으로 줄이는 방향을 검토한다.
- pandas prompt 입력은 이미 5행이므로 유지한다.

### 검증 방법

- retriever 단위 테스트에서 `preview_rows` 길이를 확인한다.
- 대표 질문 validation에서 answer 품질이 나빠지지 않는지 확인한다.

### 주의점

- `runtime_sources`와 MongoDB 저장용 원본 rows는 줄이지 않는다.
- 다운로드 기능은 full rows를 계속 받을 수 있어야 한다.

## 6. `_runtime_result_rows` 역할 정리

### 현재 상태

`17_pandas_code_executor.py`에서 `_runtime_result_rows`를 생성한다.
`23_mongodb_result_store.py`는 이 값을 `payload.result_rows`에 저장한다.

즉 완전 불필요한 필드는 아니지만, `data.rows`와 이름상 중복처럼 보인다.

### 수정 방안

- `_runtime_result_rows`를 `_full_result_rows`로 이름 변경한다.
- `data.rows`는 preview rows로 유지한다.
- `23_mongodb_result_store.py`는 `_full_result_rows`를 우선 저장하고, 없으면 `data.rows`를 fallback으로 사용한다.
- 최종 API 응답 builder는 `_full_result_rows`를 제거한다.

### 검증 방법

- pandas executor 테스트:
  - `data.rows`는 preview 제한
  - `_full_result_rows`는 전체 결과
- result store 테스트:
  - MongoDB 저장 doc의 `payload.result_rows`가 full rows를 가진다.
  - API 응답에는 `_full_result_rows`가 없다.

### 주의점

- MongoDB result loader와 download server의 path `payload.result_rows` 계약은 유지한다.

## 7. MongoDB result store 저장 구조 축소

### 현재 상태

`23_mongodb_result_store.py`는 아래를 함께 저장한다.

- `source_results`
- `runtime_sources`
- `result_rows`
- `analysis`
- `data`

여기서 `analysis.rows`, `data.rows`, `result_rows`가 일부 중복된다.

### 수정 방안

- 저장 구조를 아래처럼 정리한다.

```json
{
  "payload": {
    "request": {},
    "metadata_refs": [],
    "intent_plan": {},
    "source_results": [],
    "runtime_sources": {},
    "result_rows": [],
    "data": {
      "columns": [],
      "row_count": 0,
      "data_ref": {}
    },
    "analysis": {
      "status": "ok",
      "row_count": 0,
      "columns": [],
      "analysis_code": "...",
      "llm_generated_code": "...",
      "used_helpers": []
    }
  }
}
```

- `analysis.rows`는 저장하지 않는다.
- `data.rows`는 저장하지 않는다.
- full result는 `result_rows` 하나만 저장한다.
- source 원본은 다운로드를 위해 `runtime_sources`에만 저장한다.

### 검증 방법

- result store 테스트:
  - `payload.result_rows` 존재
  - `payload.data.rows` 없음
  - `payload.analysis.rows` 없음
  - `data_refs` path는 기존과 동일하게 동작
- result loader 테스트:
  - 이전 결과 복원 시 `data.rows` preview가 필요하면 `result_rows[:preview_limit]`로 복원한다.

### 주의점

- 기존에 저장된 result store 문서와 backward compatibility를 유지한다.
- loader는 구버전 문서의 `payload.data.rows`도 읽을 수 있어야 한다.

## 8. 답변 생성 LLM 입력 compact화

### 현재 상태

`18_answer_variables_builder.py`는 `trace.inspection` 전체를 `applied_scope_json`으로 전달한다.
이 안에는 pandas 전체 코드, helper 포함 코드, source preview, result store 정보가 들어가 token 비용이 커질 수 있다.

### 수정 방안

- 답변 LLM에는 compact scope만 전달한다.
- 포함할 정보:
  - intent summary: analysis_kind, retrieval_job_count, pandas_step_count
  - retrieval summary: source_alias, dataset_key, row_count, applied_params, pandas_filters
  - pandas summary: status, row_count, columns, used_helpers, pandas_filter_plan
  - warnings/errors
- 제외할 정보:
  - generated_code
  - effective_code_with_helpers
  - helper_sources
  - source preview rows
  - full trace/error traceback

### 검증 방법

- answer variables builder 테스트:
  - `applied_scope_json`에 pandas code가 없어야 한다.
  - row_count, columns, filters 등 답변 생성에 필요한 정보는 남아야 한다.
- playground/web 개발자모드에서는 기존처럼 pandas 진단 코드가 보여야 한다.

### 주의점

- 21번 answer message adapter의 개발자용 pandas 진단 출력은 유지한다.
- LLM 답변용 prompt 축소와 개발자 진단용 trace 보존을 분리한다.

## 9. Authoring flow 저장 trace 정책 재검토

### 현재 상태

domain/table/main variable 저장 flow는 `registration_trace.raw_text`를 MongoDB에 저장한다.
이것은 감사/추적 목적으로는 유용하지만, runtime prompt에는 불필요하다.

### 수정 방안

- 1차 수정에서는 저장 정책을 바꾸지 않는다.
- runtime metadata candidates에서만 제거한다.
- 이후 필요하면 authoring writer에 `save_registration_trace` advanced option을 추가할 수 있다.

### 검증 방법

- 저장 flow 테스트는 기존 raw text 보존 기대가 있으면 유지한다.
- data analysis prompt 후보에서는 raw text가 제거되는지 별도 테스트한다.

### 주의점

- 사용자가 최초에 말한 "txt 메타정보를 마음대로 변형해서 MongoDB에 저장하지 말라"는 원칙을 지킨다.

## 12. 완료 기준

- 의도 분석 prompt에 raw/audit 성격 metadata가 들어가지 않는다.
- `pandas_function_cases`가 function case 선택의 canonical 구조가 된다.
- preview rows는 source/trace/answer에서 과도하게 전달되지 않는다.
- MongoDB result store는 다운로드에 필요한 full rows는 보존하되 preview/data/analysis rows 중복을 줄인다.
- API 응답에는 runtime full rows와 내부 임시 필드가 남지 않는다.
- 기존 대표 질문과 token match 변형 질문이 계속 통과한다.

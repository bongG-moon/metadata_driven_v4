# 수정계획 답변품질 0703

## 목적

data analysis flow의 최종 응답을 단순 숫자/표 나열이 아니라 사람이 읽는 분석 답변 형태로 개선한다.

핵심 원칙:

- 공통 답변 프롬프트에는 특정 제조 helper나 제품 token 규칙을 하드코딩하지 않는다.
- 제조/제품/장비 ASSIGN 같은 도메인 특화 답변 지침은 별도 Text Input으로 주입한다.
- 원본 rows와 원본 숫자는 MongoDB/API 데이터에 그대로 보존하고, 메시지/웹 표시에서만 사람이 읽기 좋은 형태로 포맷한다.
- pandas 중간 결과와 function case 결과는 compact preview만 trace/analysis에 남겨 답변 근거로 사용한다.

## 구현 항목

1. pandas executor에 기록 helper 추가
   - `record_step(key, value, description="", role="")`
   - `record_function_case_result(function_name, input_text, result, description="")`
   - 기록 위치는 `analysis.step_outputs`, `analysis.function_case_results`, `trace.inspection.pandas_execution.*`

2. pandas code prompt 갱신
   - 단계형 분석에서 기준이 되는 중간 결과는 `record_step`으로 기록한다.
   - function case 결과가 답변 근거가 되면 `record_function_case_result`로 기록한다.

3. 답변 변수 확장
   - `18 답변 변수 생성기`가 `answer_context_json`을 추가 출력한다.
   - 포함 내용은 숫자 표시 정책, 중간 결과, function case 결과, 최종 결과 shape이다.

4. 답변 prompt 확장
   - `19 답변 Prompt Template`은 공통 narrative 규칙만 가진다.
   - `domain_answer_guidance` 변수를 추가해 특화 답변 지침 Text Input을 연결한다.

5. 메시지 어댑터 개선
   - 결과 테이블 숫자는 10,000 이상부터 K 단위로 표시한다.
   - 중간 결과와 function case 결과를 generic section으로 보여준다.

6. 웹 표시 개선
   - 웹 표의 기본 숫자 표시는 자동 K 기준으로 바꾼다.
   - 원본 CSV 다운로드와 Raw JSON은 원래 값을 유지한다.

## 검증 기준

- 기존 대표질문 13개가 계속 통과한다.
- token match 변형 검증이 계속 통과한다.
- `record_step`과 `record_function_case_result` 호출 결과가 payload에 남는다.
- 19번 공통 prompt에는 `match_product_tokens` 같은 특정 helper 이름이 들어가지 않는다.
- 특화 답변 지침은 Text Input으로만 전달된다.
